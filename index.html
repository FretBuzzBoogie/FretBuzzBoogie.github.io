Outbreak - Mashmo Games

Unity Engine, C#, PHP, SQL, Oculus Gear VR, Oculus GO, Oculus Rift


  • Outbreak is a single-player VR survival game. The game is based on a player collecting item-drops like weapons, bullets, interact with the environment like doors, cabinets and kill enemies to complete objectives and enjoy the story.
  • For easy creation and maintenance of the project, I created a Level Manager, Task System, Objective Manager. These components are XML based and serializable systems, to make development easy for level designers and artists.
  • Level Manager : Manages all of its sub levels, data to be set on the player on start of each sub level, the player data that needs to be saved. A level designer can easily select and edit the level data, For example: If the designer wants the player to have an AK-47 type of gun as his primary gun and the gun needs to have 50 bullets with 5 bullets in its first cartridge, the designer can easily do so from the level manager.
  • Task System : The task system includes XML pages which manages a sequence of instructions. For example : A designer can create a sequence of instructions to start/stop a sound, position the player, add/remove an object, manage the environment etc. And these sequences can be called from anywhere within the game with a unique ID.
  • Objective System : The objective system manages the range of objectives within any level. When an objective is complete, the system registers that completion and transitions the game to the next level on the completion of all objectives of that level. The objective list can be edited to be an unordered or ordered list, ordererd : an objective is complete only if the previous objective is complete.
  • All objects that are reusable in the game like item drops, weapon drops, enemies, 2D audio sources are managed by an object pool for optimization purposes to promote reusability and prevent the instantiation of an object each time it is required.
  • The entire game is interconnected by an Event System that fires an event to manage every step of the game.
  • With the use of the above features the development process is made easy for designers to add cinematic scenes, game play scenes, manage the player. Each weapon and item in the game is customizable with fire rate, features etc.
  • Created a shader for object highlighting, crosshair and environment designs.

Unity Asset : Object Pooling System - Hobby Project

Unity Engine, C#

  • Each time you create an object for use you allocate the required memory on the heap and call its constructor. When the object is not required anymore you discard the object, calling the garbage collector and thereby deallocating the memory previously acquired. These processes take time during runtime when the application needs to run efficiently.
  • The object pooling system allows for creation of the object and storing the object in a pool. When required the object is taken out of the pool and used, when the object is not required anymore it is pushed back into the pool. As the object is already created and only destroyed when the application ends, it prevents the actions of instantiation and destruction allowing for each object to be reusable.
  • If the user requires an object and the object count of the pool that manages that object is zero, the pool will automatically create the object at runtime. If the programmer understands that the application will eventually require, for example 5 objects in the future, the programmer can set the size of the pool to 5 at the start. This will allow the system to create 5 objects and push them in the pool only to be popped when required during the application's lifetime.
  • The object pool class is made to be generic and can thus be templated to be used as a pool of a specific type. This allows it to be templated of the type 'MonoBehaviour'.
  • The object pool of type MonoBehaviour can be used to store gameobjects in a pool. The templated prefab for which the gameobject should be pooled can be specified in the class constructor as well as the parent to which the pooled gameobject should be a child of.
  • A defined function on each pooled object is called when retrieved and pushed into its respective pool allowing the object to perform actions when activated / deactivated such as instantiation / release of references etc.

Multithreaded Job System - Hobby Project

C++

  • A simple job system implementing the usage of my knowledge of multithreading in C++.
  • Initialize the system on start, thereby retrieving a pointer to the system and you should destroy the system on end.
  • Jobs can be sent to be executed. Every thread on your system will be running continuously from the start. A thread will be put to sleep using a wake condition to avoid running the loop continuously if there are no jobs sent to be executed. When a job is submitted, a sleeping thread will be notified and will take it upon itself to run the job on a thread in parallel.
  • A job pool is a fixed size circular buffer that stores all the submitted jobs. The pool is responsible for managing the head and the tail. It works as a fixed size queue and with the help of mutex's it is built to be thread safe. Being thread-safe only a single thread can take the next job and a job can be put onto the buffer by preserving the head and tail position of the buffer.
  • As all threads are always running in parallel, there is no way to join the threads and the threads are only stopped on destroying the Job System. Instead, the Wait function can be used to wait for execution for jobs on all threads to be completed until the main thread proceeds.
